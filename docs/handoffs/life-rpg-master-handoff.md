## 1) Executive summary

This product is a **mobile-first Life RPG** where real-life discipline and verified effort transform the user’s:
- character
- status
- room / command center
- collection
- identity

AI acts as the **game master**, not just a chatbot.

### One-line definition
**A mobile-first Life RPG where real discipline transforms your character, status, room, collection, and identity — with AI as the game master.**

### What makes it different
This is not meant to be:
- a generic to-do app
- a basic habit tracker
- a childish RPG skin
- a pure social flex app

It is meant to be:
- a fair proof-based self-upgrade system
- a premium aspiration engine
- a visible identity transformation app
- a serious mobile-first game layer built on real life

### Core fantasy
The user should feel:
- “My real-life effort changes who I am in the game.”
- “The system is fair. If I do real work, I get real progress.”
- “My character, room, style, and collection reflect the life I’m building.”

## 2) The actual problem this app is solving

The user does not just need a task list.

The user struggles with:
- distraction
- inconsistency
- weak follow-through
- low visible payoff from real effort
- lack of motivation from generic productivity tools
- desire for tangible progress and identity change

The app solves this by combining:
- missions/tasks
- focus mode
- proof submission
- fair judgment
- rewards
- visible identity progression
- aspiration objects (style, room, cars)
- AI guidance

The product promise is not “be more productive.”
The real promise is:

**Turn real discipline into visible life progress that feels like a game.**

## 3) Product rules that should never be broken

### Non-negotiables
1. **Real effort must visibly matter.**
2. **The game must remain fair.**
3. **The premium layer must not become pay-to-win.**
4. **The app must stay mobile-first and visually premium.**
5. **Do not broaden into random side products.**
6. **Do not turn it into a childish game.**
7. **Do not let social, growth, or monetization overpower the fantasy.**
8. **Server-side authority must remain in control of progression, rewards, ownership, entitlements, and admin actions.**

### UI language rule
Never humiliate the user. Starting states can feel basic/humble, but user-facing naming must remain aspirational.

Approved status tier direction:
- Starter
- Hustle
- Rising
- Refined
- Elite

## 4) Core loop

### Primary loop
**Mission → Focus → Proof → Judge → Reward → Visible Upgrade**

### Expanded loop
1. User creates or accepts a mission
2. User enters focus/execution
3. User submits proof
4. AI or rule-based judge evaluates
5. Reward is granted
6. Skills / status / identity / room / collection progress updates
7. User sees visible progress or gets closer to visible progress
8. User is guided toward the next best step

### Product rule
Any future feature must strengthen this loop or the fantasy built on top of it.

## 20) Metrics / growth / monetization strategy

### North Star Metric
**Weekly Active Progressing Users (WAPU)**

Definition:
Users who made meaningful progression in the last 7 days, such as:
- approved proof
- mission completion
- skill/status progression
- meaningful purchase/application
- room/car/identity progression

### KPI groups
#### Activation
- signup → quick start
- quick start → first mission
- first mission → first proof
- first proof → first reward
- first reward → first purchase
- first purchase → first equip/apply

#### Retention
- D1 / D7 / D30
- comeback completion
- weekly progressing users
- chain continuation

#### Progression
- missions completed/week
- proof approval rate
- visible evolution unlock cadence
- room progression rate
- car aspiration conversion

#### Store / Economy
- browse-to-purchase
- purchase by category
- wearable equip after purchase
- room item apply after purchase
- car browse vs purchase
- coin generation vs spending

#### Premium / Revenue
- free → premium conversion
- premium feature adoption
- premium item/content conversion
- premium retention uplift

#### Growth
- invites sent
- invites accepted
- invites activated
- share/export behavior
- share → signup attribution
