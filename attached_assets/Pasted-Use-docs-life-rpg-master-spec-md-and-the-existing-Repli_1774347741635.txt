# Phase 27 — QA / Regression Foundation
Version: 1.0 | Platform: Replit | Scope: Stability, verification, release gating

## 🎯 PROJECT OVERVIEW
Build a real QA / Regression Foundation for the existing Life RPG app so core flows do not break when other systems are edited.

This phase is NOT about adding new gameplay.
This phase is about making the current product launch-safe:
- testable
- repeatable
- auditable
- releasable

The goal is to protect the product’s most important flows:
1. register / login
2. create mission
3. start focus / stop session
4. submit proof
5. judge verdict
6. reward grant
7. coin balance update
8. buy / equip / switch room / switch car / wheel / wardrobe
9. character update after verified progress

==================================================
PHASE SCOPE ONLY
==================================================

Build only the minimum QA / Regression Foundation needed to make the app safe to iterate on.

Required outputs for this phase:
1. Regression checklist
2. Smoke test list
3. Critical path pass/fail matrix
4. Bug severity rules
5. Release gate before deploy
6. Minimal automated regression coverage for high-risk logic
7. Minimal fixtures / reset tooling for repeatable QA
8. QA documentation inside the repo
9. Verification that no P0 / P1 blockers remain open at end of run

Do NOT build:
- new game systems
- new missions
- new monetization features
- new room/car/store content
- redesigns
- new admin dashboards beyond what is required for QA support
- heavy end-to-end test infrastructure if the repo does not already support it

If a full E2E framework does not already exist, do NOT force one in this phase.
Prefer:
- targeted logic tests
- lightweight API/integration tests
- manual QA sheets
- deterministic smoke scripts
- release-gate documentation

STEP 2 — DEFINE THE QA SURFACES TO PROTECT
==================================================

Create a repo-level QA inventory that clearly defines:

1. Core flow surfaces
2. Supporting dependencies
3. Data integrity checkpoints
4. User-visible failure points
5. Financial / reward integrity risk points
6. Regression-prone areas

At minimum, define these protected surfaces:

### Protected Surface A — Auth Integrity
- register with valid email/password
- reject invalid email
- reject duplicate email
- login succeeds with correct credentials
- login fails with wrong credentials
- authenticated route rejects unauthenticated access
- logout/session invalidation works if applicable

### Protected Surface B — Mission Lifecycle
- create mission works
- required fields validated
- category and proof requirements persist correctly
- mission can enter active state
- mission cannot enter invalid broken state

### Protected Surface C — Focus Session Integrity
- start session creates only one active session
- second active session is blocked if that is the product rule
- stop/end transitions session correctly
- end-early / abandon does not accidentally count as completion
- invalid stop does not corrupt session state
- pause/resume if present does not break timer/session logic

### Protected Surface D — Proof Submission Integrity
- proof can be submitted only against valid target/session/mission rules
- empty or invalid proof is rejected
- duplicate exact proof is blocked if that is the rule
- suspicious repeated proof is flagged correctly
- file/image metadata is handled safely
- proof status persists correctly

### Protected Surface E — Judge Integrity
- judge returns structured result
- provider failure falls back safely
- no verdict returns malformed or uncaught response
- follow-up does not accidentally grant reward
- reject does not grant reward
- approve grants reward exactly once
- same proof cannot be judged into multiple rewards

### Protected Surface F — Reward / Wallet Integrity
- reward amount matches expected logic
- reward transaction is atomic
- coin balance updates exactly once
- duplicate approval cannot mint duplicate rewards
- failed reward write does not partially update state
- insufficient balance checks protect purchases

### Protected Surface G — Ownership / Purchase Integrity
- buy item deducts correct coins
- already-owned item is not double-charged
- equip/switch owned item works
- buying without enough coins fails cleanly
- switching to previously owned wheel/style/environment does not charge again if that is the rule
- invalid category mismatch fails safely
- ownership records persist correctly

### Protected Surface H — Character Progress Integrity
- approved progress updates character state
- rejected proof does not
- follow-up does not prematurely advance character
- wardrobe integration still renders after progression update
- room/garage/world views reflect updated state consistently where expected

STEP 3 — CREATE THE QA FOUNDATION FILES
==================================================

Create the following docs inside the repo.

Recommended structure:

/docs/qa/
  qa-inventory.md
  regression-checklist.md
  smoke-test-list.md
  critical-path-matrix.md
  bug-severity-rules.md
  release-gate.md
  manual-qa-sheet.md
  known-risks-and-gaps.md

You may adjust names slightly if the repo already has a documentation pattern,
but keep the structure clear and easy to find.

### 3A. qa-inventory.md
Must list:
- all protected surfaces
- routes/screens/components/hooks involved
- risk level (critical / high / medium / low)
- data touched
- failure impact
- test coverage type (manual / automated / both)

### 3B. regression-checklist.md
Must include step-by-step repeatable checklist for:
- auth
- mission
- focus
- proof
- judge
- reward
- wallet
- store ownership
- character update
- room / garage / wardrobe state continuity

This checklist must be practical for a human tester.
No vague bullets.
Each item must have:
- preconditions
- action
- expected result
- actual result field
- pass/fail checkbox

### 3C. smoke-test-list.md
Create a short must-run smoke list for every release.
This is the fastest possible confidence pass.

Smoke test list must be short, brutal, and high-value.
Only include things that would seriously damage the product if broken.

### 3D. critical-path-matrix.md
Build a pass/fail matrix table for all critical launch flows.

Columns:
- flow name
- entry point
- dependencies
- expected success result
- failure impact
- severity if broken
- last verified by
- last verified at
- automated/manual
- release blocker? yes/no

### 3E. bug-severity-rules.md
Define severity levels clearly:

P0:
- app unusable
- auth fully broken
- reward duplication / coin minting
- data corruption
- crash in critical flow
- purchase/reward financial integrity broken

P1:
- major core flow blocked
- proof/judge/reward inconsistent
- session completion broken
- owned item cannot be used
- character progression wrong in core path

P2:
- degraded behavior with workaround
- non-critical incorrect UI state
- partial feature failure not blocking core loop

P3:
- cosmetic / copy / spacing / low-risk issues
- small edge-case issues with minimal user harm

Include examples specific to this app.

### 3F. release-gate.md
Create a strict release gate:
A build cannot be considered releasable unless:
1. all smoke tests pass
2. all critical-path manual checks pass
3. no open P0
4. no open P1
5. wallet/reward/purchase integrity verified
6. auth verified
7. proof/judge fallback behavior verified
8. typecheck/build/dev server still run cleanly
9. known risks explicitly documented
10. any skipped tests are called out with reason

### 3G. manual-qa-sheet.md
Create a human-friendly test execution sheet for repeating manual validation before launch.

### 3H. known-risks-and-gaps.md
Log:
- things intentionally not covered yet
- flaky areas
- temporary limitations
- manual-only checks
- future automation candidates

Preferred priority order:

Automate the highest-risk logic only.

At minimum add coverage for:

### 4A. Auth logic / route coverage
- successful register
- duplicate register rejected
- successful login
- invalid login rejected
- protected route rejects unauthenticated request

### 4B. Mission / session guardrails
- mission creation validation
- one-active-session rule if applicable
- invalid stop/end behavior handled safely

### 4C. Proof / judge guardrails
- empty proof rejected
- exact duplicate proof rejected or flagged according to current rule
- rule-based fallback returns valid structured verdict
- provider failure still returns safe structured response
- follow-up verdict does not grant reward
- reject verdict does not grant reward
- approve verdict grants reward exactly once

### 4D. Reward / wallet integrity
- reward transaction writes exactly once
- duplicate processing does not duplicate reward
- balance update equals expected amount
- rollback on simulated failure leaves no partial balance corruption

### 4E. Purchase / ownership integrity
- insufficient funds rejects purchase
- first purchase deducts coins
- already-owned behavior follows product rule
- equip/switch owned asset works
- invalid car/category/wheel/room switch fails cleanly

### 4F. Character update integrity
- approved progress updates progression state
- rejected/follow-up state does not trigger same update path
- character state remains render-safe after progression update

STEP 7 — ERROR HANDLING, SECURITY, AND EDGE CASES
==================================================

This phase must explicitly define and test the following:

## 🔐 SECURITY
- unauthenticated access to protected flows
- duplicate reward submission attempts
- duplicate purchase attempts
- invalid ownership switching
- invalid user/item/session/proof IDs
- malformed proof payloads
- race conditions where two reward/purchase actions happen near-simultaneously
- balance integrity under concurrent requests
- unsafe file metadata assumptions in proof submissions

## 🛡️ ERROR HANDLING
- judge provider unavailable
- database write partially fails
- network request retried by client
- stale UI state after purchase or reward
- session stop called twice
- proof submit called twice
- purchase endpoint hit with already-owned asset
- invalid color/wheel/environment switch target
- render-safe fallback if progression data is incomplete

## ⚠️ EDGE CASES
- user submits proof with text only
- user submits proof with image only if allowed
- user submits too-short explanation
- user resubmits after follow-up
- user abandons session then tries to submit proof
- user has exactly the minimum coins needed
- user has zero coins
- user owns multiple variants
- user progression updates while a UI screen is open
- known pre-existing non-blocking issues must be documented separately, not silently ignored
