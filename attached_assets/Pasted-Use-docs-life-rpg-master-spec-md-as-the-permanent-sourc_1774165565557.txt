==================================================
A. ANOMALY / INCIDENT MANAGEMENT LAYER
==================================================

Build a lightweight operational incident layer.

Purpose:
Give operators a place to see unusual patterns, investigate them, and track status.

At minimum, support anomaly/incident visibility for:
- signup/login failure spikes
- proof submission or approval failure spikes
- reward transaction anomalies
- upload failure spikes
- premium entitlement mismatches
- unusual store purchase anomalies
- car unlock/showcase anomalies
- room/application state anomalies
- recommendation failure/anomaly spikes
- integration/webhook issues if already present

Each incident/anomaly item should support fields like:
- type
- severity
- first seen / last seen
- current status
- affected area
- affected count if available
- short summary
- possible linked entities (user/content/system)
- owner if practical
- notes if practical

Incident statuses can include:
- new
- investigating
- mitigated
- resolved
- ignored/known issue

Requirements:
- keep this tactical and lightweight
- operators should be able to see what matters now
- use existing telemetry/admin data where possible
- do not build a giant ITSM system

==================================================
B. ADVANCED REPAIR / RECONCILIATION TOOLING
==================================================

Build safer, deeper recovery tools than Wave 1 basics.

Purpose:
Allow operators to recover broken state in high-value systems without manual DB editing.

Focus on safe recompute/reconcile actions rather than arbitrary mutation.

Support only bounded, explicit actions such as:
- recompute player progression snapshot
- reconcile wallet/reward transaction state
- reconcile owned vs equipped/displayed item state
- reconcile room/world applied state
- reconcile featured car/showcase state
- reconcile premium entitlement state
- retry failed content entitlement or unlock propagation if already applicable
- repair recommendation cache/state if already needed
- retry safe failed downstream integration jobs if already applicable

Requirements:
- each repair action must:
  - be role-gated
  - require a reason/note
  - be audit-logged
  - clearly show target scope
  - avoid unsafe bulk mutation
- prefer recomputation from trusted canonical data
- if a repair action is too risky, omit it and document it as future scope

==================================================
C. EXPERIMENT CONTROLS LITE
==================================================

Build a lightweight experimentation control layer.

Purpose:
Allow the team to safely compare a small number of product/content variants without a giant experimentation platform.

Possible experiment targets:
- onboarding copy
- next-best-action framing
- recommendation card emphasis
- store featured layout
- premium upgrade copy
- comeback mission framing
- content/event card variants
- room/car showcase CTA variants

Requirements:
- support lightweight experiment definitions
- support a small number of variants
- support assignment logic that is stable/consistent enough for the experiment
- track key outcomes using existing telemetry where possible
- allow operators to:
  - create/start/pause/stop experiment
  - inspect variant exposure
  - inspect basic outcome metrics
- keep this tactical and explainable

Do NOT build:
- a giant experimentation platform
- automated winner-pick logic
- advanced stats platform

==================================================
D. OPERATOR RUNBOOKS / OPERATIONAL GUIDANCE
==================================================

Build practical runbook surfaces inside admin.

Purpose:
Help operators know what to do when issues happen.

Add lightweight runbook / playbook support for common operational situations such as:
- proof pipeline failure spike
- reward mismatch issue
- premium entitlement mismatch
- broken live event or content pack
- broken catalog item visibility
- recommendation anomaly
- stuck player progression state
- upload/media failure cluster
- store purchase inconsistency
- car/room display mismatch

Requirements:
- runbooks should be short, practical, and tied to the real system
- each runbook can include:
  - what this issue means
  - what to check first
  - what tools to use
  - what safe actions exist
  - escalation note if appropriate
- keep it embedded or linked from relevant admin surfaces where possible
- do not build a giant documentation portal

==================================================
E. ALERTING / DIAGNOSTICS VIEWS
==================================================

Build a lightweight diagnostics surface for operators.

Purpose:
Make it easier to see failures and trace what is going wrong.

At minimum, support views for:
- recent admin/system warnings
- recent failed or degraded operations
- recent failed support/recovery actions
- recent failed integrations/webhooks if applicable
- recent failed content activations if applicable
- recent high-severity anomalies

Diagnostics should help answer:
- what failed
- when it failed
- how often it is happening
- what area it affects
- whether it is getting worse

Requirements:
- keep this concise and actionable
- use current telemetry/logging infrastructure where possible
- do not build a giant centralized logging product
- make diagnostics easy for operators to scan quickly

==================================================
F. DEEPER SUPPORT WORKFLOW
==================================================

Expand support operations from basic repair into a more useful workflow.

Purpose:
Help the team work through player issues systematically.

At minimum, support:
- support case / support issue note model lite, or an equivalent structured support thread
- status for support issues such as:
  - open
  - investigating
  - waiting
  - resolved
- link support cases to player inspector
- ability to attach notes/history to a player issue
- record what support/recovery actions were taken
- quick access to recent issues for the same player if practical

Requirements:
- keep this lightweight
- do not build a full ticketing platform
- use player-centric workflow
- keep support state admin-only
- preserve privacy and auditability

H. VALIDATION / SECURITY / AUDIT RULES
==================================================

- role permissions must be enforced server-side
- all impactful actions must be audit-logged, especially:

  - experiment start/pause/stop

- require elevated role
- require reason/note
- make the action explicit and bounded
- avoid bulk destructive behavior
