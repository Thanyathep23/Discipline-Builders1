# Phase 34 — Personalization Graph
Version: 1.0 | Platform: Replit | Scope: User state graph, personalized next actions, adaptive mission/reward framing, comeback personalization, progression pacing

## 🎯 PROJECT OVERVIEW
Build Personalization Graph v1 for the existing Life RPG app.

This phase is NOT about adding new gameplay pillars.
This phase is NOT about building a giant recommendation platform.
This phase is about making the product feel like it understands each user’s real state and can guide them differently.

The personalization graph must improve:
- next-step relevance
- mission recommendation quality
- comeback path quality
- reward/status framing relevance
- progression pacing fit
- user-specific guidance
- future retention and trust foundations

The target outcome:
1. the system can classify users by meaningful behavioral state
2. different users no longer get the exact same next-step logic
3. missions and guidance start adapting to strengths, weaknesses, and momentum
4. stalled users get different recovery treatment than active users
5. the product feels more like a personal operating system, less like a generic task app
6. the graph creates a reusable foundation for future retention, prestige, and data flywheel phases

==================================================

Before changing anything, inspect and report the current personalization-relevant system across:

- level
- xp
- skill levels/dimensions
- streak state
- trust score
- proof approval/reject/follow-up history
- mission history
- session history
- wallet/purchase history
- room/car/wardrobe ownership
- progression/character state
- any comeback/inactivity indicators
- life profile / identity profile if already present

B. Existing decision points
- mission generation/recommendation
- home screen guidance
- next-step cards
- reward surfaces
- comeback prompts
- challenge/event targeting
- progression update logic
- any admin or live ops targeting rules
- any first-use or onboarding branching

C. Existing gaps
- same missions shown to everyone
- same next-step guidance for all users
- no distinction between active vs stalled users
- no distinction between high-discipline vs low-discipline users
- no distinction between users who prefer status spending vs progress saving
- no distinction between users struggling with proof quality vs users succeeding consistently

D. Existing fairness/product risks
- personalization being too random
- personalization being too opaque
- low-quality guidance despite rich user state
- overfitting to small/noisy data
- punishing users because of temporary bad periods
- making users feel boxed into one identity
- conflict between trust engine and personalization
- conflict between live ops and personalized suggestions

E. Existing architecture risks
- signals scattered across many files
- no central user-state model
- mission logic too tightly coupled to static defaults
- no reusable recommendation layer
- personalization logic hidden in UI conditionals
- no logging for why recommendations were made

Your audit must identify:
1. what user signals already exist
2. what decisions are already partially personalized
3. what is completely generic today
4. what can be reused safely
5. what must be centralized in this phase
6. what must remain intentionally simple

Do not guess blindly.
Audit first, then build the graph.

==================================================
STEP 2 — DEFINE THE PERSONALIZATION DOCTRINE
==================================================

Create a clear Personalization Graph v1 doctrine.

The graph must satisfy all of these simultaneously:

### Goal A — Personal, Not Creepy
The app should feel relevant, not invasive.
Use behavioral/product signals, not weird overreach.

### Goal B — Useful, Not Decorative
Personalization must improve actual product value:
- better next step
- better mission fit
- better comeback path
- better progression pacing
- better emotional resonance

### Goal C — Explainable Enough
The team should be able to understand why a user got a recommendation.
Avoid black-box unpredictability.

### Goal D — Non-Punitive
A bad week should not trap a user permanently in “low potential” treatment.
The graph must be adaptive, not judgmental.

### Goal E — Compatible With Trust
Personalization should not override trust integrity.
It can shape recommendations and pacing, but must not blindly bypass proof standards.

### Goal F — Future-Ready
The graph should create foundations for:
- better retention loops
- better live ops targeting
- better status/prestige surfacing
- future identity history and data flywheel phases

STEP 5 — BUILD THE PERSONALIZED NEXT-ACTION ENGINE
==================================================

Create a lightweight next-action engine powered by the user graph.

This engine must choose or prioritize the next best action for a user based on current state.

At minimum support recommendations such as:
- create one manageable mission
- do one focused session
- resubmit stronger proof
- recover streak gently
- save for a first status item
- use owned assets / equip / personalize identity
- complete one comeback challenge
- push progression in a weak dimension
- reduce friction and restart small

The engine must account for:
- current momentum
- trust/proof quality
- progression stage
- economy state
- identity motivation state
- recent failures/successes

For each next-action recommendation define:
- action id
- user states it fits
- priority logic
- confidence
- copy framing angle
- what screen/flow it should point to
- what makes it a bad recommendation

Rules:
- do not recommend overwhelming tasks to stalled users
- do not recommend spend-heavy actions to broke/new users unless motivational and realistic
- do not recommend aggressive proof-heavy actions to users struggling with trust/follow-up
- recommendations should feel helpful and achievable

STEP 15 — SECURITY, FAIRNESS, ERROR HANDLING, EDGE CASES
==================================================

## 🔐 SECURITY / PRIVACY
Explicitly protect against:
- exposing sensitive internal state labels to the wrong surfaces
- storing creepy/unnecessary personal inferences
- recommendation routes leaking privileged data
- unsafe admin access to detailed behavioral state without justification

## ⚖️ FAIRNESS / PRODUCT SAFETY
Explicitly review for:
- users not being trapped in negative states too long
- personalization not bypassing trust requirements
- personalization not unfairly prioritizing spend-heavy actions
- comeback users not being over-rewarded unfairly versus consistent users
- user history not being treated as destiny

## 🛡️ ERROR HANDLING
Handle:
- missing graph state
- partial data availability
- old users with incomplete telemetry
- graph version mismatch
- recommendation generation failure
- conflicting states
- stale state snapshot
- home screen fallback if personalization fails

## ⚠️ EDGE CASES
Document and handle:
- brand new user with little data
- highly active user with weak proof quality
- high trust user with temporary slump
- low activity but high spender
- no first purchase but strong progression
- status-engaged user with low mission completion
- user returning after long absence
- user with mixed signals across dimensions
- recommendation that points to unavailable flow
- live ops event recommendation conflicting with comeback recommendation
