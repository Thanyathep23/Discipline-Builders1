# Phase 32 — Launch Package
Version: 1.0 | Platform: Replit | Scope: App store positioning, launch messaging, onboarding copy, screenshot system, landing/marketing assets, launch readiness package

## 🎯 PROJECT OVERVIEW
Build the first real Launch Package for the existing Life RPG app.

This phase is NOT about adding new gameplay.
This phase is NOT about redesigning the product.
This phase is about making the product legible, desirable, and launch-ready from the outside.

The goal is to turn the current app into something that:
- people immediately understand
- people want to try
- people can trust
- people can explain to others
- the founder can launch with confidence

This phase must package the product clearly across:
- app store positioning
- launch narrative
- screenshot/storyboard system
- title / subtitle / keywords
- onboarding messaging
- tagline options
- short and long descriptions
- launch checklists
- first audience definition
- founder/operator launch-ready documents

The target outcome:
1. the app has a clear public-facing positioning
2. store assets and launch copy are coherent
3. onboarding communicates the value instantly
4. launch-day messaging is ready
5. the team has a real launch checklist
6. the product is easier to present, market, and test with real users

==================================================
PHASE SCOPE ONLY
==================================================

Build only the minimum strong launch package needed for a serious initial launch.

Required outputs for this phase:
1. launch audit
2. positioning doctrine
3. ICP / first audience definition
4. core messaging hierarchy

If store assets do not exist, create the system/specification for them and implement only what is necessary inside the current repo.

STEP 1 — AUDIT FIRST
==================================================

Before changing anything, inspect and report the current public-facing product presentation across:

A. Product understanding
- what the app currently appears to be from the UI
- whether the value proposition is obvious
- whether the app looks like a generic productivity app
- whether the “proof-based identity evolution” concept is visible enough

B. Existing launch surfaces
- app icon if present
- splash/loading text
- onboarding flow
- welcome screen
- auth entry screens
- home/hero surfaces
- any landing page or marketing page
- any app store draft copy
- any screenshots/promotional images already present

C. Existing messaging
- current titles
- current subtitles
- current CTAs
- current onboarding copy
- current feature labels
- current explanations of missions / proof / rewards / identity / room / wardrobe / garage / world

D. Current positioning risks
- product feels too complicated
- fantasy is not obvious
- app sounds like just another to-do app
- reward/status value unclear
- trust/proof system unclear
- room/car/avatar/store feels superficial or confusing
- launch messaging lacks emotional hook

E. Current audience clarity
- who the product seems built for
- who should be the first launch audience
- which audience is too broad for v1
- what promise is most believable for launch

Your audit must identify:
1. what public-facing surfaces already exist
2. what message is currently strongest
3. what message is confusing or weak
4. what must be clarified before launch
5. what should be emphasized vs deprioritized in launch messaging

Do not guess blindly.
Audit first, then package the launch.

==================================================
STEP 2 — DEFINE THE POSITIONING DOCTRINE
==================================================

Create a clear launch-stage positioning doctrine.

The launch package must satisfy all of these simultaneously:

### Goal A — Instant Clarity
A new person should understand the app quickly.
They should not need a long explanation to grasp:
- what it is
- why it is different
- why it matters

### Goal B — Strong Category Framing
The app must not sound like a generic task app.
It must feel like a system where:
real progress
→ proof
→ reward
→ identity/status/world evolution

### Goal C — Emotional Pull
The launch message must sell not just productivity,
but transformation, momentum, status, and visible self-progress.

### Goal D — Launch-Stage Credibility
Promises must be ambitious but believable.
Do NOT promise superhuman outcomes or vague hype.

### Goal E — Focused Audience
The launch package must be designed around a clear first audience, not “everyone.”

### Goal F — Reusability
The launch package should generate reusable assets:
- store copy
- screenshots
- onboarding
- landing hero copy
- launch post copy
- founder pitch

==================================================
STEP 3 — DEFINE THE FIRST AUDIENCE / ICP
==================================================

Define the launch-stage ideal first audience.

At minimum document:
- who this product is best for first
- who it is not for yet
- what pain/state they are in
- what transformation they want
- what emotional hook matters most to them
- what feature they care about first
- what objection they are most likely to have

Examples of possible audience frames to evaluate:
- ambitious self-improvement users
- students who want discipline structure
- early-career builders/operators
- gym/productivity/status-oriented users
- people who fail at habit apps because nothing feels real
- users who want visible momentum, not just checkbox completion

Choose a primary launch audience and optionally one secondary audience only.
Do NOT target everyone.

For each audience define:
- pain
- desire
- trigger to try
- reason to stay
- likely objection
- best messaging angle

STEP 4 — BUILD THE MESSAGING HIERARCHY
==================================================

Create the messaging stack from top to bottom.

At minimum define:

### Layer A — Core Promise
One sentence answer to:
“What is this app?”

### Layer B — Why It’s Different
One sentence answer to:
“Why not use a normal habit/to-do app?”

### Layer C — Emotional Payoff
One sentence answer to:
“What does the user get to feel?”

### Layer D — Functional Proof
One sentence answer to:
“How does it actually work?”

### Layer E — Status/Identity Hook
One sentence answer to:
“Why does the world/avatar/status layer matter?”

Then turn this into:
- 3 positioning versions
- 5 tagline options
- 3 short elevator pitches
- 1 recommended launch narrative

All messaging must stay coherent with the product:
proof-based progress, rewards, coin economy, evolving identity, room/world/status surfaces.

==================================================
STEP 15 — SECURITY, TRUST, ERROR HANDLING, EDGE CASES
==================================================

## 🔐 TRUST / CLAIM SAFETY
- do not promise features not fully working
- do not overclaim outcomes
- do not market unverified systems as magic
- do not misrepresent proof/judge fairness
- do not fake scarcity or fake prestige messaging

## 🛡️ ERROR HANDLING
- if some launch surfaces do not yet exist, create docs/spec instead of fake implementation
- if screenshots require assets not ready yet, specify them clearly
- if onboarding copy depends on screens not finalized, keep copy modular
- if landing page surface is partial, prioritize doc/package quality first

## ⚠️ EDGE CASES
- product is powerful but too complex for first impression
- user only cares about productivity, not status/world
- user only cares about status/world, not proof
- launch audience misreads the app as a game only
- launch audience misreads the app as a strict work tracker only
- screenshots overemphasize one part and bury the core loop
- onboarding explains too much too early
- taglines sound cool but unclear
- store copy sounds unique but not credible

==================================================

==================================================

Important implementation rules:

1. Do NOT redesign the entire app for launch.
2. Do NOT create giant marketing infrastructure.
3. Focus on clarity, coherence, and launch readiness.
4. Prefer strong messaging systems over flashy but vague copy.
5. Keep all claims grounded in the actual product.
6. Make the product easy to understand in seconds.
7. Optimize for launch-stage conversion and clarity.
8. Keep the deliverables easy for the founder to use immediately.
