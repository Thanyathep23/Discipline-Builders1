Build only the smallest high-impact controlled community layer.

==================================================

==================================================

Build only:

1. Invite-only accountability circles / pods

4. Small social visibility for meaningful progress

Do NOT build:
- public social feed
- comments feed

- open follower graph

- global leaderboard
- public-by-default profiles
- creator marketplace

- full community platform
- redesign of the whole app

STEP 1 — AUDIT FIRST
==================================================

Before changing anything:
1. inspect the current shareable layer, invite/referral system, identity/profile surfaces, titles/badges, live content systems, admin tools, and telemetry
2. identify what already exists that can support controlled community/accountability
3. identify the minimum set of social/accountability features needed to create retention and meaning without toxicity
4. implement only the minimum systems needed
5. avoid broad refactors

==================================================
A. INVITE-ONLY ACCOUNTABILITY CIRCLES / PODS
==================================================

Add a lightweight private social layer based on small invite-only groups.

Purpose:
Help users stay consistent by sharing progress with a small trusted set of people, not the whole internet.

Build:
- circles / pods / groups with small size limits
- invite-only membership
- owner/admin role for the circle
- basic member state
- ability to join via invite link/code if feasible
- circle visibility limited to members only

Suggested size:
- small, intimate groups only (for example 2-8 people max)

Requirements:
- circles are private by default
- membership is explicit and controlled
- no random discovery of users or groups
- each circle should feel like an accountability pod, not a social feed
- preserve privacy and user control

Possible data shown inside a circle:
- active title
- current arc
- selected progress cards
- selected challenge participation
- optional recent wins

Do NOT expose:
- sensitive profile answers
- raw proof files
- private mission notes
- exact financial/private data
- anything not explicitly chosen for visibility

==================================================
B. OPT-IN PRESTIGE SHOWCASE SURFACES
==================================================

Add an opt-in way for users to present a curated version of their progress to others.

Purpose:
Let users feel proud and visible without making the system shallow or toxic.

Build:
- public-to-circle or shared-to-circle prestige snapshot
- toggleable profile visibility settings for shareable identity elements
- selective visibility of:
  - active title
  - current arc
  - top strengths
  - selected badges/trophies
  - selected share cards
  - current mastery/prestige markers if they exist

Requirements:
- all visibility must be opt-in
- users should control what is shown
- showcase should feel premium and aspirational
- no public-by-default profile exposure
- no sensitive personal answers by default

==================================================
C. LIGHTWEIGHT SHARED CHALLENGES / GROUP ACCOUNTABILITY
==================================================

Add a small shared challenge layer for circles.

Purpose:
Create community-driven motivation without requiring synchronous play or heavy social features.

Examples:
- Focus Weekend Challenge
- 3-Day Momentum Reset
- Trading Review Sprint
- Recovery Week
- Learning Sprint

Build:
- circle-level challenge participation
- visible progress at a lightweight level
- simple participation/completion state
- challenge completion moments for the group
- optional circle rewards or symbolic unlocks if appropriate

Requirements:
- shared challenges should reuse existing mission/pack/event systems where possible
- do not build separate giant challenge logic if existing systems can power it
- keep challenge visibility simple and encouraging
- avoid shame-heavy or public-failure mechanics
- allow async participation

==================================================
D. SMALL SOCIAL VISIBILITY FOR MEANINGFUL PROGRESS
==================================================

Add only a minimal social visibility layer for meaningful moments.

Examples of visible events inside a circle:
- title unlocked
- badge earned
- chain completed
- comeback mission completed
- mastery milestone reached
- live event/challenge completed
- selected share/result card posted to circle

Rules:
- only meaningful moments should surface
- do not spam small/noisy events
- users should be able to control or mute what is shared
- visibility should be encouragement-oriented, not addictive feed design

Possible UI surfaces:
- circle activity panel
- recent meaningful wins
- accountability snapshot
- current shared challenge progress
- “celebrate” reaction or lightweight acknowledgement if appropriate

Keep this minimal and tasteful.

==================================================
E. LOW-TOXICITY COMMUNITY DESIGN RULES
==================================================

This product must avoid the worst patterns of social apps.

Requirements:
- no global ranking by default
- no public humiliation mechanics
- no visible failure wall
- no manipulative engagement spam
- no vanity-first design that destroys the self-improvement mission
- social visibility should emphasize:
  - effort
  - consistency
  - meaningful wins
  - support
  - accountability
not shallow comparison

If comparison is used at all, keep it:
- private
- limited
- opt-in
- contextual
- non-toxic

==================================================
F. MINIMAL MODERATION / SAFETY CONTROLS
==================================================

Because this introduces user-to-user surfaces, add minimum safety controls.

Build only lightweight essentials:
- leave circle
- remove member (owner/admin only)
- basic report/flag action for inappropriate shared content if any shared surface exists
- admin/internal visibility for moderation events if feasible
- audit logging for circle management actions

Requirements:
- preserve privacy
- preserve ownership/authorization
- keep moderation tools minimal but real
- do not build a full trust-and-safety suite

==================================================

- not feed-addicted
- should feel like a selective accountability network

- do not trust client-side visibility settings alone
