# Phase 33 — Trust Engine v2
Version: 1.0 | Platform: Replit | Scope: Fair verdicts, anti-gaming, confidence routing, explainability, trust logging, low-cost decision safety

## 🎯 PROJECT OVERVIEW
Build Trust Engine v2 for the existing Life RPG app.

This phase is NOT about adding new gameplay.
This phase is NOT about building a giant AI moderation platform.
This phase is about making proof judgment more trustworthy, more consistent, more explainable, and more resilient.

The trust engine must improve:
- verdict fairness
- consistency across similar cases
- protection against low-effort or fake submissions
- confidence-aware routing
- low-cost handling for easy cases
- escalation for ambiguous/high-risk cases
- logging for future learning and audits

The target outcome:
1. trivial or obviously invalid proofs are handled safely and cheaply
2. high-confidence clean proofs are handled consistently
3. ambiguous proofs do not get over-rewarded
4. suspicious patterns are flagged instead of blindly approved
5. verdicts become easier to explain and audit
6. the system gains a reusable trust foundation for future personalization and data flywheel phases

STEP 2 — DEFINE THE TRUST DOCTRINE
==================================================

Create a clear v2 trust doctrine.

The trust engine must satisfy all of these simultaneously:

### Goal A — Fairness
Users doing real work should not feel randomly punished.
The same type of submission should produce roughly similar outcomes.

### Goal B — Friction With Purpose
Not every proof should be instantly approved.
Low-quality or ambiguous submissions should be challenged appropriately.

### Goal C — Anti-Gaming
The engine must resist:
- empty effort
- duplicated effort
- template spam
- obvious low-signal submissions
- suspicious repeated patterns

### Goal D — Explainability
Every non-trivial verdict should have usable reasons:
- why it passed
- why it got follow-up
- why it was rejected
- what is missing if more proof is needed

### Goal E — Cost Discipline
Easy cases should not consume expensive processing unnecessarily.
Ambiguous or risky cases deserve stronger handling.

### Goal F — Future Extensibility
The trust layer should create foundations for:
- future personalization
- future appeals/review
- future data flywheel
- future smarter routing

STEP 3 — STANDARDIZE THE VERDICT MODEL
==================================================

Create a clear structured verdict schema and migrate existing trust/judge outputs toward it.

At minimum define verdict states such as:
- approved
- followup_required
- rejected
- flagged_for_review if needed and feasible
- system_error / retry_safe if needed

For each verdict define:
- meaning
- reward eligibility
- whether progression may advance
- whether user feedback is shown
- whether manual review may be required
- whether it is trust-sensitive

Create a structured verdict payload shape such as:

- verdict
- confidence_level
- confidence_score if appropriate
- trust_risk_level
- reward_decision
- progression_decision
- primary_reasons[]
- missing_requirements[]
- suspicious_signals[]
- escalation_recommended
- provider_used
- rules_triggered
- evaluation_version

Rules:
1. avoid vague freeform-only responses
2. keep output structured and auditable
3. make it safe for logging and analytics
4. do not expose unsafe internal prompts or secrets to clients
5. ensure reward and progression are downstream consequences of structured verdicts, not fuzzy side effects

==================================================
STEP 4 — BUILD THE TRUST ROUTING MODEL
==================================================

Create a lightweight trust routing model.

At minimum define these case classes:

### Class A — Easy Clean Case
Examples:
- valid structured submission
- no duplicate/suspicious signals
- strong evidence
- low ambiguity

Handling:
- cheapest valid route
- high-confidence fast decision
- structured verdict
- audit log retained

### Class B — Ambiguous Case
Examples:
- some effort present but insufficient detail
- unclear evidence
- borderline quality
- incomplete explanation
- weak image/text linkage

Handling:
- follow-up preferred over blind reject
- request better evidence
- stronger reasoning output
- no reward until resolved

### Class C — Risky / Suspicious Case
Examples:
- near-duplicate proof
- repeated low-effort patterns
- suspicious timing or volume
- conflicting evidence
- abnormal reuse behavior

Handling:
- stricter scrutiny
- stronger rule enforcement
- reward blocked unless high confidence
- escalation / flagging path if appropriate

### Class D — System / Provider Failure Case
Examples:
- provider timeout
- malformed response
- unavailable provider
- empty model output

Handling:
- safe fallback
- no accidental reward
- retry-safe or follow-up-safe response
- trust log entry

For each class define:
- routing rule
- provider or rule path
- allowed verdicts
- escalation behavior
- reward safety constraints
- logging requirements

==================================================
STEP 5 — BUILD THE CONFIDENCE MODEL
==================================================

Create a practical confidence system.

This does NOT need advanced ML.
It needs a bounded launch-safe confidence layer.

At minimum define:
- confidence_low
- confidence_medium
- confidence_high

Confidence should consider factors such as:
- completeness of submission
- clarity of explanation
- evidence quality/signal
- consistency with expected proof type
- duplicate/suspicious signals
- provider certainty if available
- fallback/rule-based confidence
- user history only if already available and safe to use

Rules:
- low confidence should never mint full reward casually
- medium confidence may produce follow-up more often
- high confidence may approve only if no serious risk signal exists
- confidence must not be confused with trustworthiness of the user alone
- current user trust score, if used, must be only one signal, not absolute truth

Create a documented confidence rubric and central config.

==================================================

Create a lightweight anti-gaming layer using current available data and logic.

At minimum evaluate signals such as:
- exact duplicate proof

- repeated boilerplate explanations
- repeated suspicious submission timing
- low-information submissions
- inconsistent proof-to-mission mapping
- same content reused across submissions
- abnormal approval-seeking spam behavior
- repeated follow-up-triggering behavior
- repeated suspicious failures after rewards if trackable

For each signal define:
- signal name
- trigger logic
- severity
- whether it blocks approval directly
- whether it lowers confidence
- whether it recommends follow-up
- whether it recommends escalation
- false-positive risk

Do NOT create aggressive punishment systems in this phase.
Focus on:
- safe detection
- confidence adjustment
- escalation
- protection against obvious abuse

STEP 14 — SECURITY, FAIRNESS, ERROR HANDLING, EDGE CASES
==================================================

## 🔐 SECURITY
Explicitly protect against:
- duplicate reward minting through verdict confusion
- provider malformed output causing accidental approval
- suspicious repeated submissions bypassing checks
- unsafe admin review powers
- inconsistent verdict-to-reward mapping
- route-level replay or duplicate processing

## ⚖️ FAIRNESS
Explicitly review for:
- similar submissions treated similarly
- real effort not punished too harshly
- follow-up used instead of reject when ambiguity is fixable
- user history not overriding real evidence unfairly
- confidence and suspicion used carefully, not as blunt punishment

## 🛡️ ERROR HANDLING
Handle:
- provider timeout
- empty model result
- malformed structured result
- fallback failure
- missing proof metadata
- evaluation version mismatch
- logging failure
- reward/progression write blocked after verdict

## ⚠️ EDGE CASES
Document and handle:
- user submits solid text-only proof if allowed
- user submits image with weak explanation
- duplicate content with slightly changed wording
- high trust user submits weak evidence
- low trust user submits strong evidence
- follow-up then strong resubmission
- provider disagreement across retries
- suspicious signal appears but evidence is otherwise strong
- approved verdict returned but downstream reward write fails
- progression update lags after correct approval
