A. SKILL GROWTH SUGGESTIONS
==================================================

Add actionable growth suggestions to the Skill screen.

For each of the 6 core skills:
- Focus
- Discipline
- Sleep
- Fitness
- Learning
- Trading

show:
- why the skill is currently at this level/trend
- what is helping it
- what is hurting it
- 2-4 practical next actions to improve it

Requirements:
- suggestions must be based on actual user data where possible
- suggestions must feel specific, not generic filler
- suggestions should connect to mission categories already in the app
- suggestions should help the user understand how to level the skill in real life

Examples:
- Focus: reduce blocked attempts, complete one uninterrupted 25-minute session, finish a deep work mission with strong proof
- Discipline: accept fewer missions but complete more, reduce abandonment, maintain consistency
- Learning: submit better summaries, complete study missions with takeaways, increase proof quality
- Trading: complete verified chart review, submit reasoning, do backtesting/journaling missions with evidence

Implementation:
- add backend service or route logic that derives suggestions from:
  - skill trends
  - recent missions
  - completed vs abandoned sessions
  - blocked attempts
  - proof quality
  - mission categories
  - arc if useful
- if data is sparse, use strong rule-based defaults
- expose suggestions in the skill summary endpoint
- render them clearly in the Skill screen UI

==================================================
B. FILE UPLOAD PROOF TYPES
==================================================

Extend the proof system so missions can use file upload as a first-class proof type.

Requirements:
- support proof file uploads for relevant missions
- connect uploaded files to proof submissions
- support at least:
  - image files
  - PDF/doc-like files if current upload system can support safely
  - optional fallback to image-only if broader file support is risky
- store file metadata securely
- validate file type and file size
- return JSON errors on invalid uploads
- preserve the existing proof -> judge -> reward flow

Rules:
- file uploads must remain ownership-safe
- user can only access their own uploaded proof files
- proof uploads must not break current text/link proof flows
- do not make file upload mandatory for all missions
- only require files when the mission proof requirement calls for it

Mission examples where file upload matters:
- Trading review -> upload journal/export/screenshot bundle
- Learning mission -> upload notes or worksheet
- Deep work/output mission -> upload file artifact or document
- Fitness mission -> usually image/log, not document-heavy

Implementation:
- reuse existing upload/storage system if present
- if no safe generalized upload system exists, implement the smallest secure version needed
- persist uploaded files in proof file records
- ensure AI judge / rule-based judge can see file metadata and use it in evaluation
- if file content cannot be deeply parsed yet, still use metadata + presence + mission fit
