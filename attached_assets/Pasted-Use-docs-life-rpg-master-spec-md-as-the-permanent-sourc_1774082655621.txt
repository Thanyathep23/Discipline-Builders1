STEP 1 — AUDIT FIRST

==================================================
A. RELEASE READINESS / CRITICAL FLOW HARDENING
==================================================

Ensure the main user journeys are launch-safe.

Critical flows to harden and verify:
- signup / login / session recovery
- onboarding / profile progression
- user mission flow
- AI mission flow
- focus session flow
- proof submission flow
- proof upload flow
- reward/wallet flow
- inventory / marketplace / equip flow
- room/world display flow
- premium entitlement checks
- circle/community flow if already live
- integrations / calendar / export/import if already live

Requirements:
- improve error recovery where still weak
- improve partial-state recovery where needed
- improve fallback behavior when AI/integration services are unavailable
- ensure empty states and degraded states do not strand the user
- preserve JSON-only APIs and ownership checks

==================================================
B. RELIABILITY / RECOVERY / FAIL-SAFE BEHAVIOR
==================================================

Add or improve lightweight reliability and recovery controls.

Examples:
- job or process-safe retry strategy where appropriate
- idempotency or duplicate-protection on sensitive actions where still missing
- safe handling of interrupted proof upload / reward finalization / chain advancement / entitlement grant
- graceful fallback when AI, OCR, PDF extraction, calendar, webhook, or payment/premium components fail
- recovery UI or admin repair paths for corrupted partial state

Requirements:
- main flow should not be fragile when downstream services fail
- failure in non-critical subsystems must not break the entire user experience
- keep reliability mechanisms lightweight and practical
- no giant distributed systems redesign

==================================================
C. PERFORMANCE / SCALE READINESS
==================================================

Add only the minimum performance/scale readiness needed for launch confidence.

Audit and improve where necessary:
- obvious N+1 or repeated unnecessary queries
- expensive dashboard/profile/world queries
- large payload responses
- heavy image/file handling bottlenecks
- admin pages with unbounded lists
- marketplace/inventory/world rendering hot spots
- telemetry/admin/live-ops queries that may grow badly

Requirements:
- add pagination or query limits where needed
- add indexes where clearly necessary
- reduce obvious waste
- preserve correctness
- do not over-engineer premature scaling infrastructure

If load testing is feasible, add a lightweight sanity check.
If not, add operational notes and clear bottleneck awareness.

==================================================
D. SECURITY / FRAUD / ABUSE HARDENING
==================================================

Strengthen launch-critical security and abuse resistance.

Audit and improve:
- auth/session protection
- proof/reward abuse
- invite/referral abuse
- circle/community misuse
- premium entitlement abuse
- double-spend or duplicate purchase paths
- upload abuse
- webhook / integration abuse if applicable
- admin route protection
- dangerous missing validation paths

Requirements:
- preserve server-side authority for progression, rewards, entitlements, ownership, and display state
- improve rate limiting where still weak
- improve audit logging where needed
- add lightweight fraud/abuse detection or safeguards where clearly valuable
- keep changes focused and practical

==================================================
E. OPERATIONAL SUPPORT / DIAGNOSTICS / RECOVERY TOOLS LITE
==================================================

Add only minimal internal tools needed to operate post-launch.

Possible examples:
- failed job/integration delivery visibility
- stuck proof/review/reward state inspector
- user state repair tool for critical corruption cases
- premium entitlement repair/re-sync tool
- content/event activation sanity view
- error/event diagnostics summary
- “safe replay/retry” actions for selected flows
- basic user support admin view for launch support

Requirements:
- admin-only
- all actions audit-logged
- no dangerous bulk mutation tools unless absolutely necessary
- keep tools tactical, not enterprise-bloated

==================================================
F. ROLLOUT / KILL-SWITCH / SAFETY CONTROLS
==================================================

Add lightweight release control mechanisms.

Examples:
- feature flags or kill-switches for high-risk systems:
  - AI mission generation
  - OCR/PDF extraction
  - premium purchases
  - share/invite/community surfaces
  - integrations/webhooks
  - live events / packs
- safe degrade modes
- rollout flags for new features
- admin/internal emergency disable controls

Requirements:
- keep controls lightweight
- admin/internal only
- allow product to stay usable if a subsystem must be disabled
- preserve trust and continuity

==================================================
G. POST-LAUNCH OPERATIONS / RUNBOOK LITE
==================================================

Add lightweight operational documentation and internal guidance.

Create or improve practical docs such as:
- launch checklist
- critical flows checklist
- incident response notes
- “what to check first” for common issues
- recovery steps for stuck user states
- support/debug notes for admins
- post-launch monitoring checklist

Requirements:
- keep docs practical and short
- based on the real system as implemented
- no vague theoretical documentation

- upload failure spikes

- circle/invite abuse spikes
- integration/webhook failure spikes

Requirements:

- do not build a giant observability platform
