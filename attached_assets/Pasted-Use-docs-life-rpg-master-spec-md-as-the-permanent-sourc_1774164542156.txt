==================================================
A. CONTENT OPS CONSOLE
==================================================

Build the core admin console for shipping and managing content.

Purpose:
Allow the team to create, inspect, schedule, activate, pause, and archive content without code edits.

At minimum, support these content entities where applicable:
- mission packs
- challenge packs
- comeback packs
- live events
- limited-time drops
- seasonal/cycle content
- premium content packs
- featured collections or themed content if already supported

Each content item should support operational fields such as:
- name
- type
- short description
- status
- start/end time if relevant
- targeting/eligibility summary if relevant
- featured state if relevant
- reward or incentive summary if relevant
- linked catalog/store items if relevant
- created/updated timestamps

Required statuses:
- draft
- scheduled
- active
- paused
- expired
- archived

Admin must be able to:
- create content
- edit content
- schedule content
- activate/deactivate content
- archive content
- inspect linked items/missions if already practical

Requirements:
- keep this tactical, not a giant CMS
- server-side validation required
- important writes must be audit-logged
- do not require code edits for basic activation/deactivation after this wave

==================================================
B. ECONOMY CONSOLE
==================================================

Build a lightweight but real Economy Console.

Purpose:
Give visibility into coin generation, spending, pricing pressure, and pacing health.

At minimum, show:
- total coin generation over recent windows
- total coin spend over recent windows
- sink/source ratio
- top reward sources
- top spend categories
- purchases by category:
  - wearables
  - room/workspace
  - cars
  - premium cosmetics/content if relevant
- recent large reward or spend anomalies if practical
- average/median wallet balance if already easy
- recent price-sensitive item performance if practical

Admin should be able to inspect:
- item purchase volume
- item conversion by category
- affordability friction signals if already practical
- potentially underpriced or overpriced item candidates (lightweight, heuristic is fine)

Admin controls can include only lightweight tuning if already safe:
- adjust item price
- adjust level unlock threshold
- activate/deactivate featured push
- change item category visibility
- inspect suspicious economy behavior

Requirements:
- keep reward/economy authority server-side
- keep this operational, not a giant financial warehouse
- all impactful admin mutations must be audit-logged
- preserve existing store/economy behavior for users

==================================================
C. RECOMMENDATION / AI OPS CONSOLE
==================================================

Build a lightweight ops surface for recommendation and AI-guided systems.

Purpose:
Allow the team to see whether the recommendation layer is useful and to tune it safely.

At minimum, support visibility into:
- next-best-action recommendations
- mission recommendations
- smart merchandising/store recommendations
- recovery/comeback recommendations if applicable

Admin should be able to inspect:
- recommendation type
- recommendation target surface
- impression count
- click/use/accept rate if already tracked
- purchase/completion follow-through if already practical
- low-performing recommendation groups
- recommendation reasoning summary if available
- whether recommendation came from rule-based or AI-assisted logic if relevant

Admin controls can include:
- enable/disable specific recommendation surfaces
- adjust weighting/priorities for recommendation categories
- reduce store-push aggressiveness if too strong
- increase recovery recommendations if user base is struggling
- inspect why a recommendation was selected for a given player if already practical

Requirements:
- do not build a giant ML ops platform
- keep recommendation logic inspectable and explainable
- admin changes must be bounded and audit-logged
- preserve trust and product clarity

==================================================
D. FEATURE FLAGS / KILL SWITCHES
==================================================

Build proper Wave 2 feature controls for high-risk systems.

Purpose:
Allow the team to disable risky systems safely without taking down the whole product.

At minimum, support kill-switch or feature-flag control for:
- AI mission generation
- OCR / file extraction if present
- recommendation surfaces
- live events visibility
- content pack visibility
- premium offer surfaces
- car collection surfaces if needed
- photo mode if needed
- integrations/webhooks if already present
- community/circle surfaces if already present

Requirements:
- controls must be admin-only
- system should degrade safely when a feature is disabled
- users should not get broken dead-end UX when a flag is off
- flag changes should be audit-logged
- keep implementation lightweight and reliable

Support at minimum:
- on/off
- optional rollout status / label if easy
- note/reason for change if practical

==================================================
E. LIVE OPS CONTROLS
==================================================

Build lightweight controls specifically for running the product as a living game.

Purpose:
Allow operators to control what is active now.

At minimum, support:
- currently active events list
- currently active content packs list
- featured collection / featured store section controls if applicable
- visibility of active premium packs if relevant
- active seasonal/cycle content if already supported
- “what is live right now” summary

Admin should be able to:
- activate/deactivate live content
- pause broken live content
- inspect timing windows
- inspect basic performance summary for active content
- avoid overlapping/contradictory active states where practical

Requirements:
- keep this practical
- use the content/event models already in the system
- do not build a giant live-service orchestration engine

==================================================
F. DEEPER FUNNEL / TELEMETRY VIEWS
==================================================

Build a more useful internal funnel console beyond the high-level dashboard.

At minimum, support visibility into these funnels:
1. Signup -> Quick Start completion
2. Quick Start -> first mission
3. First mission -> first proof
4. First proof -> first reward
5. First reward -> first store visit
6. First store visit -> first purchase
7. First purchase -> first equip/apply
8. Invite received -> signup -> activated user
9. Free -> premium
10. Comeback surface shown -> comeback mission accepted/completed if applicable

If practical, segment by:
- direct vs invite
- free vs premium
- new vs returning
- active vs comeback users

Requirements:
- keep views tactical and legible
- do not build a giant analytics warehouse
- use existing telemetry where possible
- allow operators to identify obvious drop-off points quickly

==================================================
H. VALIDATION / SECURITY / AUDIT RULES
==================================================

Requirements:
- all Wave 2 admin routes must be protected server-side
- role permissions must be enforced server-side
- all impactful actions must be audit-logged, especially:
  - content activation/deactivation
  - price changes
  - unlock threshold changes
  - feature flag changes
  - recommendation weighting or visibility changes
- preserve JSON-only admin APIs
- do not expose sensitive raw user data unnecessarily
- preserve user-facing ownership, entitlement, progression, and economy rules

If a Wave 2 action could destabilize the product:
- require higher admin role
- require reason/note
- make the UI explicit and careful
- do not silently mutate major state
