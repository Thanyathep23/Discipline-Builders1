# Phase 28 — Economy Tuning v1
Version: 1.0 | Platform: Replit | Scope: Coin economy, pricing, reward pacing, affordability, progression balance

## 🎯 PROJECT OVERVIEW
Build the first real economy tuning layer for the existing Life RPG app.

This phase is NOT about adding new fantasy systems.
This phase is about making the current product economy feel:
- fair
- motivating
- aspirational
- sustainable
- hard to exploit
- ready for real users

The economy already has active surfaces:
- missions
- proof/judge/reward
- coin balance
- wardrobe purchases
- car purchases
- wheel customization
- room environment purchases
- progression-linked identity/status rewards

This phase must turn that into a coherent economy.

The target outcome:
1. users feel meaningful progress early
2. premium/status items still feel aspirational
3. users cannot inflate too fast
4. sinks and sources are balanced
5. the economy can be tuned from centralized config instead of scattered magic numbers
6. the repo gains economy documentation, tuning logic, and simulation tooling

==================================================
STEP 2 — DEFINE THE ECONOMY DESIGN GOALS
==================================================

Create a clear design doctrine for v1 economy.

The economy must satisfy all of these simultaneously:

### Goal A — Early Progress Feels Real
A new user should feel first ownership momentum quickly.
They must not be stuck earning coins with nothing meaningful to buy.

### Goal B — Premium Status Still Feels Earned
Cars, premium outfits, better rooms, and prestige-feeling assets must retain aspirational value.
They cannot become trivial after a few sessions.

### Goal C — Identity Progress and Economic Progress Must Align
If character progression and store/status progression move at radically different speeds, the fantasy breaks.
The user should feel:
"I am becoming more disciplined, and my world/status is evolving with me."

### Goal D — The Economy Must Be Tunable
The team must be able to adjust:
- reward values
- price values
- level gates
- rarity bands
- affordability targets
without hunting through random files.

### Goal E — The Economy Must Resist Inflation
Highly active users should advance faster, but not so fast that room/car/status rewards collapse in meaning.
There must be guardrails for runaway earnings.

### Goal F — The Economy Must Support Launch Readiness
The economy should be understandable, documented, and reviewable before release.

STEP 5 — DEFINE REWARD BANDS
==================================================

Centralize reward tuning into a real reward model.

At minimum define reward bands by:
- mission difficulty / effort
- mission impact level
- proof quality / judge result
- session duration or validated effort band if applicable
- streak multiplier or consistency bonus
- any trust/fairness modifiers that already exist

Build a reward doctrine like this:
- trivial / low-effort approved work should not mint high rewards
- meaningful consistent work should feel rewarded
- follow-up_required never grants full reward
- rejected proof grants zero reward
- approved proof grants exactly one reward path
- bonuses should feel good but not explode the economy

Create a centralized reward band table with fields like:
- band id
- label
- intended use
- min reward
- target reward
- max reward
- allowed modifiers
- notes

Do NOT create an overcomplicated formula if the product does not need it yet.
Prefer a clear bounded system over fancy math.

STEP 7 — SET AFFORDABILITY TARGETS
==================================================

Define practical affordability targets for real player progression.

At minimum define targets for:

### New user / Day 1
- first meaningful item
- first upgrade moment
- how many approved missions/sessions it should roughly take

### Engaged user / Day 7
- one or more wardrobe upgrades
- maybe one stronger status item
- progress toward car/room aspiration
- no feeling of stagnation

### Active user / Day 30
- clear ownership history
- meaningful room/car/status progression
- not total completion
- still has aspirational goals left

Create explicit target questions such as:
- How fast should a new user earn their first non-free wearable?
- How long should the first car take?
- How long should a premium room take?
- Can a highly active user buy too many rooms in one week?
- Can a casual user buy nothing meaningful for too long?

Output an affordability table that answers these directly.

STEP 14 — SECURITY, ERROR HANDLING, EDGE CASES
==================================================

## 🔐 SECURITY
Explicitly review and preserve or strengthen:
- duplicate reward mint prevention
- duplicate purchase prevention
- insufficient funds protection
- already-owned purchase protection
- invalid item/category/environment/wheel purchase protection
- race conditions around wallet updates
- transaction integrity for reward and spend paths
- admin/manual grant misuse risk if such routes exist

## 🛡️ ERROR HANDLING
Define and validate behavior for:
- failed reward transaction
- failed purchase transaction
- inconsistent wallet state
- item price/config missing
- level gate mismatch
- stale client price display vs server-authoritative price
- simulation/config loading failure
- partially migrated old economy values

## ⚠️ EDGE CASES
Handle/document:
- user has exactly enough coins
- user has zero coins
- user buys multiple low-tier items quickly
- user saves for premium item without intermediate purchases
- user has high activity but many follow-ups instead of approvals
- user hits level gate before price gate
- user hits price gate before level gate
- ownership switch should not charge when already owned if that is the rule
- wheel/environment re-selection should not double charge
- economy config drift between backend and frontend display
