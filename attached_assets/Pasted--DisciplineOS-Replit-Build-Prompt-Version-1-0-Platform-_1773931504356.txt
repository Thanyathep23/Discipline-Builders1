# DisciplineOS — Replit Build Prompt
Version: 1.0 | Platform: Replit | Stack: Next.js + React + TypeScript + Tailwind + Supabase + Chrome Extension (Manifest V3) + OpenAI API

Build a production-minded MVP called **DisciplineOS**: a gamified self-discipline platform that combines daily mission planning, focus mode, website blocking, time tracking, proof-based task completion, AI verification, and a reward economy. The product should feel like a **dark premium command center** with a **serious discipline tone** and a **light RPG reward layer**.

This is **not** a generic to-do app. The core product promise is:
- help users decide what matters today,
- make distraction harder during focus time,
- verify whether work was actually done,
- reward meaningful progress fairly,
- and discourage low-effort completion or cheating.

Prioritize a **working end-to-end MVP** over visual over-engineering. Avoid fake demo logic. Build real flows with real persistence.

---

## 🎯 PROJECT OVERVIEW

Create a multi-user web app plus a Chrome extension.

### Main user flow
1. User signs up / signs in.
2. User creates daily missions with duration, priority, purpose, and proof requirements.
3. User starts a focus session for a mission.
4. Web app enters focus mode and paired browser extension blocks configured distracting websites during the active session.
5. User finishes the session and submits proof:
   - image / screenshot
   - text summary
   - file upload
   - external link
6. AI Judge reviews the proof and either:
   - approves and grants reward,
   - partially approves with reduced reward,
   - rejects and asks follow-up questions,
   - or flags the submission for manual admin review.
7. User sees coins, streaks, level progress, penalties, and daily/weekly analytics.
8. Admin can manage users, shop items, flagged submissions, and system rules.

### Product principles
- No reward should be granted from timer completion alone.
- Important work should pay more than trivial work.
- Weak proof should get low reward or no reward.
- Repeated attempts to bypass focus mode should create penalties.
- Sleep and recovery should matter, but in v1 sleep is manual entry.
- Streaks matter, but never more than actual work quality.

### App architecture
- Use Next.js App Router.
- Use TypeScript everywhere.
- Use Tailwind for styling.
- Use Supabase for Auth, Postgres, Storage, and Row Level Security.
- Use server-side routes / server actions for all privileged operations.
- Never trust client-side completion, reward, or proof verdict logic.

### State management
- Use server state as source of truth.
- Use lightweight client state only for UI state, timers, filters, and modals.
- Prefer React context or Zustand only where necessary.
- Do not create a complex global store unless truly needed.

### Data flow
1. User action in UI
2. Zod validation
3. Server route / server action
4. Supabase read/write with RLS
5. AI judge route for proof evaluation
6. Reward engine computes result server-side
7. Persist audit logs
8. Revalidate UI

### Extension flow
- Extension must authenticate to the web app using a short-lived token exchange.
- During active focus sessions, extension receives blocking rules for the logged-in user.
- When user opens blocked sites, extension:
  - blocks navigation or redirects to local blocked page,
  - shows reason and active mission,
  - records blocked attempt,
  - syncs attempt to backend when online.
- If extension is offline, queue events locally and sync later.

## 🔐 AUTHENTICATION & SECURITY

### Auth method
- Use Supabase Auth with email + password for MVP.
- Support password reset.
- Require verified session for all user actions.
- Admin area must be protected by role-based authorization.

### Roles
- `user`
- `admin`
- `system`

Notes:
- `system` is not a human login role. It represents server-side AI judge / reward engine actions.
- Admin pages must never rely on hidden UI alone. Enforce authorization server-side and in DB policies.

### Protected routes
- `/dashboard`
- `/missions/**`
- `/focus/**`
- `/proofs/**`
- `/rewards/**`
- `/analytics`
- `/sleep`
- `/settings/**`
- `/admin/**` (admin only)

### Security rules
- Never expose Supabase service role key to client.
- All reward calculations happen server-side only.
- All AI verdict persistence happens server-side only.
- File uploads must use signed upload flow and restricted buckets.
- Validate MIME type, file size, and extension on upload.
- Sanitize all free text fields before storage and rendering.
- Escape user-generated content in UI.
- Use Zod validation on every form and API route.
- Rate limit:
  - auth attempts
  - proof submissions
  - AI judge requests
  - extension heartbeat / sync endpoints
- Use server timestamps, not client timestamps, for sensitive logs.
- Maintain audit logs for:
  - reward grants
  - reward reversals
  - proof verdicts
  - admin actions
  - strictness rule changes
- Prevent replay or duplicate proof submissions for the same session if already finalized.
- Prevent direct client manipulation of:
  - mission completion status
  - coin balance
  - streak counters
  - penalty counters
  - admin flags

### Anti-cheat / fraud prevention
- Do not auto-approve based only on elapsed time.
- Track session heartbeats from web app and extension where applicable.
- Mark sessions as low-confidence if:
  - no proof submitted,
  - proof text is too short or generic,
  - uploaded content appears unrelated,
  - link is invalid or inaccessible,
  - session had many blocked-site attempts,
  - extension was disconnected during a strict focus block.
- Maintain `trust_score` per user that adjusts follow-up strictness over time.
- Repeated suspicious behavior should increase manual review likelihood.

## 🔒 ROW LEVEL SECURITY (RLS) RULES

Enable RLS on all user-data tables.

### General rule
Users can only read and write their own records unless role = admin.

### Example policy intent
- `profiles`: user can read/update own profile, admin can read all
- `missions`: user owns own missions
- `focus_sessions`: user owns own sessions
- `blocked_domains`: user owns own domain rules
- `blocked_attempts`: user can read own attempts, system insert via server
- `proof_submissions`: user owns own submissions
- `proof_files`: user owns own files
- `reward_transactions`: user can read own ledger only
- `wallets`: user can read own wallet; write only via server logic
- `sleep_logs`: user owns own logs
- `audit_logs`: only admin can read; user cannot tamper

### Critical rule
Do **not** allow direct client updates to:
- wallet balances
- admin flags
- AI review rows
- final proof verdict rows
- audit logs

Use server-side service-role operations only for these sensitive writes.

- mission priority
- mission impact
- target duration
- session metrics
- blocked attempt count
- user strictness mode

- proof file metadata

### AI Judge outputs (strict JSON)

- verdict

- quality_score
- relevance_score
- duration_plausibility_score
- needs_followup
- followup_questions[]
- reward_multiplier
- penalty_recommendation
- flags[]

### AI Judge behavior
- Be strict but fair.

### Model routing
- Use a lighter model for first-pass review.
- Use a stronger model only when:
  - confidence is low,
  - proof is ambiguous,
  - user disputes the verdict,
  - evidence is complex.

- duration weight
- proof quality
- proof confidence
- strictness modifier
- trust score modifier
- streak bonus
- sleep bonus
- penalty deductions

### Strictness modes
**Strict**
- fewer allowed pauses
- harsher penalty for blocked attempts
- higher proof threshold
- higher reward ceiling for strong proof

**Balanced**
- default mode
- moderate proof threshold
- moderate penalties

**Custom**
- user-adjustable within safe limits
- cannot reduce all friction to zero

### Penalty logic
- blocked attempt spikes
- abandoning sessions repeatedly
- suspicious duplicate proofs
- repeated low-effort submissions
- bypass-like behavior

### Streak logic
A streak should increment only when a meaningful day criterion is met, such as:
- at least one approved mission above minimum quality threshold
or
- configurable daily discipline score passes threshold

Do not increment streak from trivial activity alone.

## 🛡️ ERROR HANDLING

Handle these clearly:

### Network errors
- show retry UI
- do not lose draft proof text
- queue extension sync when offline

### Validation errors
- field-level messages
- prevent invalid durations, dates, file types, URLs

### AI errors
- fallback status: `manual_review` or `retry_pending`
- never silently grant reward if judge fails
- log invalid AI outputs

### Upload errors
- show exact reason
- keep submission draft intact
- allow retry without rewriting everything

### Auth errors
- expired session redirect to login
- safe post-login return flow

### Permission errors
- return 403 UI for admin-only routes
- avoid leaking protected data

### Not found
- useful 404 pages for mission, proof, shop item, admin entities

### Auth & Roles

- admin routes are protected
- normal users cannot access admin data

### Missions
- user can create, edit, archive missions
- mission status updates correctly
- daily missions display properly

- only one active focus session at a time
- pauses, resumes, and completion work
- blocked attempt counts are recorded

### Extension
- extension can connect to authenticated user
- blocked domains sync correctly
- blocked site redirect works during active focus session
- blocked attempts sync back to backend

### Proof & AI Judge
- user can submit proof with text + file and/or link
- AI returns structured review
- follow-up questions work
- final verdict persists correctly
- reward is not granted without valid review

### Rewards
- approved proof creates reward transaction
- rejected proof creates no reward
- penalties can reduce net reward
- shop purchase deducts coins correctly
- inventory updates correctly

### Sleep
- user can log sleep
- sleep bonus works within defined cap
- sleep does not overpower work rewards

### Analytics
- dashboard shows real user data
- weekly analytics are persisted and derived from actual records
- planned vs completed and blocked attempts are visible

### Security
- RLS prevents cross-user access
- service-role-only writes are respected for sensitive data
- file uploads are restricted and validated
- audit logs exist for sensitive actions

### Quality
- no critical TypeScript errors
- no broken protected routes
- no hardcoded fake balances or verdicts
