==================================================
A. CURATED MISSION PACKS / CHALLENGE PACKS
==================================================

Add a lightweight curated content layer.

Purpose:
Allow the team/admin to create and activate themed mission packs or challenge packs without changing core logic.

Examples:
- 7-Day Focus Reset Pack
- Trading Review Sprint
- Deep Work Rebuild Pack
- Recovery Week Pack
- Proof Quality Upgrade Pack
- Comeback Challenge Pack

Requirements:
- packs can contain a curated set of missions/chains/templates
- packs can target a skill, arc, or player state
- packs can be manually activated/deactivated
- packs can be limited-time or evergreen
- packs can be assigned eligibility rules if feasible
- accepted pack missions should still use the existing mission/proof/reward flow
- keep the implementation data-driven and lightweight

Do NOT build a giant narrative campaign engine.
Keep packs tactical and reusable.

==================================================
B. LIVE EVENT / LIMITED-TIME CONTENT LITE
==================================================

Add a lightweight live event layer.

Purpose:
Create time-bounded motivation without building a huge event system.

Examples:
- weekend challenge
- comeback event
- focus week
- mastery sprint
- end-of-month push
- arc-specific event

Requirements:
- event has:
  - name
  - description
  - start/end window
  - content source (pack, chain, missions, bonuses, or visibility)
  - target user state if applicable
- event visibility must be time-aware
- event completion can grant titles, badges, symbolic assets, or limited bonus rewards
- event logic must remain server-side
- event content must feel curated and intentional

Rules:
- keep events lightweight
- do not overwhelm the user with many simultaneous events
- events should enhance pacing, not create chaos

==================================================
C. CONTENT SCHEDULING / ACTIVATION CONTROLS
==================================================

Add lightweight internal controls for activating content.

Requirements:
- admin/internal can:
  - activate/deactivate mission packs
  - activate/deactivate events
  - set start/end windows
  - set visibility/eligibility constraints if already easy to support
- content status should be inspectable:
  - draft
  - active
  - scheduled
  - expired
  - archived
- content should not require code changes for every activation
- use existing admin/tuning surfaces where possible
- keep schema and tooling minimal and additive

==================================================
D. LIGHTWEIGHT EXPERIMENTATION / VARIANT TESTING
==================================================

Add a small experimentation layer for key content surfaces.

Purpose:
Allow the team to test what content framing or pacing works better without building a massive experimentation platform.

Examples of experimentable targets:
- AI mission framing
- comeback mission copy
- pack CTA wording
- event card positioning
- reward explanation copy
- mission difficulty framing

Requirements:
- support lightweight variants
- assign a variant consistently per user when needed
- track variant exposure and key outcomes using existing telemetry if possible
- keep this simple and tactical
- do not build a general-purpose experimentation platform

Suggested outcomes to track:
- mission accept rate
- completion rate
- proof submission rate
- reward conversion
- share/export click if relevant
- return/comeback completion

==================================================

Add only the minimum internal UI/tools needed.

Possible admin/internal tools:

- live event list + editor lite

- content performance summary

Requirements:
- admin-only
- use current dark premium command-center style
- keep interfaces tactical, clear, and lightweight
- do not build a giant CMS

==================================================
F. USER-FACING SURFACES
==================================================

Add only minimal user-facing surfaces needed for live content.

Examples:
- active event banner/card
- active challenge pack card
- limited-time mission group section
- event completion result card
- pack progress surface if appropriate

Requirements:
- keep it elegant and non-spammy
- content should feel like special opportunities, not clutter
- use existing mission/arc/identity vocabulary
- preserve privacy and seriousness of the product

==================================================
G. CONTENT ELIGIBILITY / TARGETING LITE
==================================================

If practical and lightweight, allow content to target user states such as:
- current arc
- weak skill
- comeback state
- mastery/prestige readiness
- proof quality issues
- recent consistency

Rules:
- keep targeting simple and explainable
- avoid black-box complexity
- if OpenAI is unavailable, targeting should still work with rule-based logic
- do not create hard-to-debug hidden behavior

==================================================
H. BACKEND / SECURITY RULES
==================================================

- all content activation and scheduling must be server-side
- all admin/content-ops routes must require admin authorization
- all experiments/variants must be assigned server-side or via safe internal logic
- preserve ownership checks
- preserve JSON-only APIs
- keep schema changes additive and minimal
- do not expose internal experiment state unnecessarily to normal users
- preserve existing mission/proof/reward flows
