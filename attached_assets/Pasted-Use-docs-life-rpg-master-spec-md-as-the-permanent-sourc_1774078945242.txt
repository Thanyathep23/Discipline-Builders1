Goal:
Implement Phase 18 — World / Room / Lifestyle Presentation Layer Lite.

Current context:
- Core loop works
- Life RPG layer works
- Proof intelligence works
- Adaptive challenge, quest chains, rarity, identity layer, telemetry, growth, endgame, live-ops, community/accountability, integrations, and economy/marketplace systems are complete
- The next highest-value gap is presentation: users can now earn, unlock, buy, equip, and progress, but the world/room/lifestyle layer where those items and statuses visibly live is still too shallow
- The product now needs a premium presentation layer that makes progress and ownership feel spatial, visual, and emotionally rewarding

Your task:
Build only the smallest high-impact world / room / lifestyle presentation layer.

==================================================
PHASE 18 SCOPE ONLY
==================================================

Build only:

1. Command Center / Room presentation layer
2. Lifestyle / prestige display surfaces
3. Slot-based item display / placement lite
4. Theme / environment progression lite
5. Profile-world linkage
6. Minimal admin/internal controls if needed

Do NOT build:
- full 3D world
- open-world navigation
- giant house builder
- freeform furniture physics
- advanced decorating simulator
- player-to-player housing visits as a large social system
- MMO-style world instances
- redesign of the whole app

==================================================
STEP 1 — AUDIT FIRST
==================================================

Before changing anything:
1. inspect the current marketplace, inventory, equipped items, titles, badges, prestige markers, profile surfaces, showcase/share cards, and any existing “command center” or room-like UI
2. identify what assets already exist that can be visually displayed
3. identify the minimum spatial/presentation layer needed to make ownership and progression feel embodied
4. implement only the minimum systems needed
5. avoid broad refactors

==================================================
A. COMMAND CENTER / ROOM PRESENTATION LAYER
==================================================

Build a lightweight “world” surface centered around a premium command center / room / base view.

Purpose:
Give the user a place where progression lives visually.

Requirements:
- a primary room/command center screen or section
- visual zones or sections where unlocked/purchased items can appear
- clear distinction between:
  - equipped/displayed
  - owned but not displayed
  - locked/unavailable
- room should reflect identity/progression, not just act as static wallpaper
- use current dark premium / command-center style

Suggested room zones:
- title / identity header zone
- current arc / season / prestige zone
- trophy/badge display zone
- lifestyle/room upgrade zone
- featured asset / centerpiece zone
- mastery/prestige marker zone

Rules:
- keep the room elegant and clean
- do not try to build a complex simulation
- this is a presentation layer first, not a sandbox builder

==================================================
B. LIFESTYLE / PRESTIGE DISPLAY SURFACES
==================================================

Add a lightweight lifestyle/prestige display layer.

Purpose:
Show what the user has become, not just what they own.

Possible elements to display:
- active title
- current arc and arc stage
- prestige/ascension markers
- mastery markers
- featured badges/trophies
- symbolic assets from marketplace
- current season/cycle progress
- featured rare/limited items

Requirements:
- these displays should feel aspirational and status-relevant
- they should pull from real progression data
- they should visually reward sustained effort
- keep it premium and believable, not gaudy

==================================================
C. SLOT-BASED ITEM DISPLAY / PLACEMENT LITE
==================================================

Add a simple display/placement system.

Purpose:
Allow owned assets to be shown in a world-like space without building a full decorator.

Requirements:
- use slot-based placement, not freeform drag-anywhere physics
- support a small number of display slots, for example:
  - centerpiece slot
  - badge/trophy shelf slot(s)
  - room theme slot
  - title display slot
  - prestige marker slot
  - featured asset slot
- users can assign eligible owned items to supported slots
- users can remove/swap displayed items
- equipped/displayed state remains server-side

Rules:
- keep placement simple, deterministic, and stable
- not every item type needs a slot
- some items may be “profile only,” some “room only,” some both
- avoid overbuilding a housing editor

==================================================
D. THEME / ENVIRONMENT PROGRESSION LITE
==================================================

Add lightweight room/environment progression.

Purpose:
Make the world itself evolve as the user progresses.

Examples:
- command center tiers
- room themes
- prestige room accents
- mastery visual markers
- arc-themed environment states
- event/season overlays if already easy to connect

Requirements:
- environment/theme changes should come from:
  - purchases
  - milestones
  - prestige/mastery
  - special events/cycles if already available
- room themes should feel like a serious lifestyle/status upgrade
- keep implementation lightweight and modular
- users can select active theme if they own more than one

Do NOT build a giant skin engine.
Keep it tactical and elegant.

==================================================
E. PROFILE-WORLD LINKAGE
==================================================

Connect the room/world view to the existing profile/identity layer.

Requirements:
- profile/character summary should reflect the current displayed world state where relevant
- world/room should reinforce:
  - active title
  - current arc
  - strengths
  - prestige/mastery
  - equipped symbolic items
- shareable/profile showcase surfaces should be able to draw from the room/world layer if useful
- the product should feel like one connected identity system, not disconnected screens

==================================================
F. USER-FACING UI MVP
==================================================

Build only minimal UI needed.

Possible surfaces:
- Room / Command Center screen
- Display slots editor lite
- Theme selector
- Featured asset selector
- World-linked profile summary improvements

Requirements:
- dark premium
- command-center prestige
- serious, aspirational, luxurious, not childish
- visually rewarding but not cluttered
- prioritize readability and high-end feel
- make it screenshot-worthy where possible

==================================================
G. ADMIN / INTERNAL CONTROLS LITE
==================================================

Add only minimal internal controls if needed.

Possible controls:
- activate/deactivate themes or room items
- classify item display eligibility
- mark items as room-displayable, profile-displayable, or both
- inspect which world assets are being used most
- control limited/exclusive environment items if needed

Requirements:
- admin-only
- use existing admin/tuning surfaces where possible
- keep tools tactical and lightweight

==================================================
H. BACKEND / SECURITY RULES
==================================================

- all ownership, equip/display, and theme activation checks must be server-side
- preserve JSON-only APIs
- do not trust client-provided ownership or display state
- keep schema changes additive and minimal
- preserve existing mission/proof/reward/community/marketplace flows
- keep world/display state auditable where useful

==================================================
I. ACCEPTANCE CRITERIA
==================================================

This phase is complete only if:

1. a command center / room presentation layer exists
2. owned/equipped/displayed world assets can be shown clearly
3. users can assign eligible owned items to simple display slots
4. lifestyle/prestige/progression is visually represented in the room/world layer
5. room themes or environment progression exists in a lightweight form
6. profile and world layers feel linked
7. server-side validation protects ownership and display state
8. existing mission/proof/reward/marketplace flow still works
9. no oversized 3D/housing-simulator scope was added
