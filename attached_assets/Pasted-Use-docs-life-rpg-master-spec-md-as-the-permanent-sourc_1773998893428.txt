==================================================
A. REWARD ECONOMY BALANCE
==================================================

Tune the reward economy so rewards feel motivating but controlled.

Goals:
- easy missions should not pay too much
- hard and meaningful missions should feel rewarding
- chain bonuses should feel special but not inflate too fast
- rarity bonuses should feel exciting but controlled
- proof quality should still matter more than pure repetition
- streak/consistency should help, but not dominate quality

Requirements:
- keep all reward logic server-side only
- preserve explainability of bonuses
- keep changes data-driven where possible
- avoid runaway inflation
- do not redesign the whole economy

Suggested areas to tune:
- base mission reward scaling
- rarity bonus values
- chain completion bonus values
- adaptive challenge bonus values
- soft caps or dampening when users chain too many easy wins

==================================================
B. DAILY RE-ENGAGEMENT / COMEBACK MISSIONS
==================================================

Add a small retention layer for users who return daily or come back after falling off.

Build:
1. Daily re-engagement logic
2. Comeback missions after inactivity/slump

Requirements:
- if a user is active and consistent, show a “best next mission” or “today’s recommended mission”
- if a user has dropped off for some days, generate a comeback mission that is easier, lower-friction, and psychologically safe
- comeback missions should feel like re-entry, not punishment
- missions should be aligned to current arc and weak skills where possible

Examples:
- “Rebuild momentum with one 15-minute focus block”
- “Recovery step: upload one small piece of trading review”
- “Reset chain: complete one light discipline action today”

Rules:
- do not shame the user
- do not immediately assign hard missions after a slump
- use gentle but meaningful re-entry design

==================================================
C. ANTI-BURNOUT / PACING GUARDRAILS
==================================================

Improve pacing so the game remains engaging without overwhelming the user.

Requirements:
- detect overload or negative momentum
- reduce mission pressure when the user is struggling
- avoid stacking too many hard missions at once
- avoid repeated high-friction proof requirements when the user is already overloaded
- allow challenge to rise again when momentum improves

Use existing signals where possible:
- completion rate
- abandonment
- blocked attempts
- proof quality
- recent mission volume
- current arc
- recent failure patterns

Keep this lightweight and practical.

==================================================
D. BETTER MISSION PACING
==================================================

Improve how many and what type of AI missions are generated.

Goals:
- avoid mission spam
- make each AI mission feel more intentional
- maintain a healthy mix of:
  - maintenance missions
  - growth missions
  - recovery/reset missions
  - stretch/challenge missions

Requirements:
- users should not get too many simultaneous AI missions
- hard missions should feel earned
- recommended mission selection should feel smart
- use rule-based fallback if OpenAI is unavailable
