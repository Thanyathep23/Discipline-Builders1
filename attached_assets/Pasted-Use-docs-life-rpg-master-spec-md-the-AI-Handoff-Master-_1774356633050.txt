# Phase 29 — Metrics / Decision Dashboard v1
Version: 1.0 | Platform: Replit | Scope: Event taxonomy, KPI definitions, dashboard foundation, decision instrumentation

## 🎯 PROJECT OVERVIEW
Build the first real Metrics / Decision Dashboard foundation for the existing Life RPG app.

This phase is NOT about adding new gameplay.
This phase is NOT about building an enterprise BI stack.
This phase is about making the product measurable so the team can make launch and post-launch decisions using real data.

The dashboard foundation must answer questions like:
- Are users activating?
- Where do they drop off?
- Are proofs being submitted?
- Are proofs being approved fairly?
- Are rewards healthy?
- Is the economy inflating?
- Are users buying status assets?
- Are room / wardrobe / garage systems actually being used?
- Is the core loop working?
- What should the founder look at every day?

The target outcome:
1. a clean event taxonomy exists
2. the highest-value product events are instrumented
3. the highest-value KPIs are defined and computable
4. a founder-facing dashboard exists in a practical v1 form
5. product decisions can be made from metrics instead of intuition alone

==================================================
PHASE SCOPE ONLY
==================================================

Build only the minimum strong metrics foundation needed for launch readiness and early live decision-making.

Required outputs for this phase:
1. metrics audit
2. event taxonomy
3. KPI dictionary
4. dashboard metric definitions
5. instrumentation for core flow events
6. economy metric instrumentation
7. dashboard data aggregation layer
8. founder/admin dashboard v1
9. alert/watchlist metric definitions
10. repo documentation for metrics and decisions

Do NOT build:
- a huge analytics warehouse
- third-party enterprise analytics stack unless already present and trivial to use
- complex attribution modeling beyond current scope
- ML forecasting
- vanity charts with no product value
- marketing analytics platform
- A/B testing engine
- retention CRM automation
- live ops engine
- new gameplay features

If the app already has telemetry or audit logs, reuse and extend them.
Do NOT duplicate event systems unnecessarily.

STEP 2 — DEFINE THE DECISION QUESTIONS
==================================================

Build this phase around concrete founder/product questions.

The dashboard must answer, at minimum:

### Acquisition / Activation
- How many new users registered today / this week?
- How many new users created a first mission?
- How many new users started a first focus session?
- How many new users submitted a first proof?
- How many new users reached first approved proof?
- How long does it take to reach first meaningful progress?

### Core Loop Health
- How many missions are created per day?
- How many sessions are started / stopped?
- What % of started sessions become proof submissions?
- What % of proof submissions become approved / follow-up / rejected?
- What % of approved proofs generate reward successfully?
- Is the core loop leaking at any step?

### Trust / Judge Health
- Approval rate
- Follow-up rate
- Reject rate
- Provider fallback rate
- judge failure/error rate
- suspicious duplicate proof signals
- dispute-prone surfaces if measurable

### Economy Health
- Coins minted per day
- Coins spent per day
- Net wallet delta
- Average reward by type/category
- Purchase conversion rate
- Time to first purchase
- Most purchased categories
- Inflation warning signals
- abnormal mint/spend events

### Status / Identity Engagement
- Wardrobe usage
- Equip rate
- Room switch rate
- Car switch rate
- Wheel switch rate
- Character progression advancement rate
- Milestone/visual progression frequency

### Retention / Ongoing Use
- D1 retention
- D7 retention
- returning active users
- streak participation if measurable
- repeat proof submitters
- repeat purchasers
- users who stall after first failure / rejection

The dashboard must serve decisions, not curiosity.

==================================================
STEP 12 — SECURITY, PRIVACY, ERROR HANDLING, EDGE CASES
==================================================

## 🔐 SECURITY / PRIVACY
- do not expose sensitive user data in dashboard payloads
- keep dashboard access admin/founder only
- no raw proof content in metrics views unless already explicitly allowed
- no secret tokens / private metadata in telemetry payloads
- financial and trust-sensitive metrics must come from backend-confirmed events

## 🛡️ ERROR HANDLING
- dashboard still renders if one query fails
- clear empty state if data is insufficient
- clear partial-data state if some metrics are not ready
- avoid hard crashes on missing events
- fail safely on bad date range input

## ⚠️ EDGE CASES
- zero-data system before launch
- very small sample sizes
- users with purchases but no recent sessions
- old data with inconsistent instrumentation
- events arriving from retried requests
- item switch without new purchase
- follow-up then later approval path
- reward granted after delayed processing
- missing retention data for immature cohorts
