==================================================
A. ASSET APPLICATION MODEL
==================================================

Build a clear model for how owned items are actually applied in the product.

Purpose:
Make item ownership meaningful by distinguishing between:
- owned
- eligible to equip/display
- currently equipped
- currently displayed
- inactive / stored
- unlock-only / passive
- profile-only / room-only / status-only / multi-surface

Each item should be able to declare where it can apply, for example:
- character/status
- room/world
- profile/showcase
- command center
- prestige display
- identity header
- shared showcase if allowed

Requirements:
- keep the model data-driven
- not every item needs to be applicable everywhere
- some items are display-only
- some are equip-only
- some are passive markers/unlocks
- some are multi-surface

Do not overcomplicate item semantics.
Keep it tactical and extensible.

==================================================
B. EQUIP / UNEQUIP / SET-ACTIVE FLOWS
==================================================

Add the real item application actions.

Support at minimum:
- equip
- unequip
- set as active
- swap active item
- display in eligible slot
- remove from display slot

Requirements:
- all equip/application logic must be server-side validated
- users cannot equip items they do not own
- users cannot equip items into invalid slot/surface combinations
- if an item type is unique-active (such as active title, active watch, active theme), enforce exclusivity
- if multiple slots of a category are allowed, make the rule explicit and minimal
- keep state transitions predictable and auditable

Rules:
- some items should be one-active-only
- some items may be displayable in multiple surfaces if the product model allows it
- titles, badges, trophies, accessories, vehicles, room upgrades, and prestige markers may all behave differently
- avoid building a giant RPG inventory engine

==================================================
C. SLOT MAPPING / DISPLAY SURFACE SYSTEM
==================================================

Add a clear slot system that maps item types to where they can appear.

Purpose:
Ensure items connect cleanly to the character, world, and profile.

Suggested slot families:
- character/status slots
  - active title
  - watch/accessory
  - jewelry/accent
  - prestige marker
  - specialist identity slot
- room/world slots
  - centerpiece
  - trophy shelf
  - room theme
  - featured lifestyle item
  - vehicle display
  - command center upgrade
- profile/showcase slots
  - featured badge
  - featured title
  - featured asset
  - featured milestone marker

Requirements:
- define which item categories can go into which slot families
- make slot compatibility server-side enforced
- keep slot count limited and elegant
- avoid overbuilding a full decorator or wardrobe manager
- if some items affect a layer without visible slotting, support that as passive application

==================================================
D. CHARACTER / STATUS CONNECTION
==================================================

Connect equipped/applied items directly into the Phase 21 character evolution system.

Requirements:
- equipped items should affect the computed visual/status representation where appropriate
- watches/jewelry/accessories should influence accessory or lifestyle layer
- vehicles or high-status items may affect lifestyle/status backdrop or display state
- prestige items may affect elite overlays or prestige accents
- specialist items may affect specialist identity layer
- do not let items completely override the progression logic — they should enhance, not fake the underlying status

Rules:
- real progression still matters more than purchased signals
- items should amplify or refine the visual fantasy, not fabricate mastery the user has not earned
- keep the result psychologically satisfying and fair

==================================================
E. ROOM / WORLD CONNECTION
==================================================

Connect equipped/displayed items into the command center / room / world layer.

Requirements:
- room-displayable items should actually appear in the room/world presentation
- active room themes should update the environment state
- trophies/badges/featured assets should appear in their intended zones
- premium/exclusive/limited items should visually stand out, but tastefully
- world state should update when the user changes displayed items

Rules:
- keep world rendering deterministic
- avoid giant re-render complexity
- prioritize elegance and consistency over flashy excess

==================================================
F. PROFILE / SHOWCASE CONNECTION
==================================================

Connect item application to identity/profile surfaces.

Requirements:
- profile/character summary should reflect active/equipped identity items where relevant
- featured titles/badges/assets should show correctly
- share/showcase surfaces should be able to draw from active applied items
- “equipped” and “displayed” states should feel consistent across profile, world, and status surfaces

Do not redesign the share layer.
Add only the linkage required.

==================================================
G. INVENTORY / ITEM DETAIL / FEEDBACK UX
==================================================

Upgrade the inventory and item detail UX so users understand what an item does and where it applies.

Requirements:
- inventory should clearly show:
  - owned
  - equipped
  - displayed
  - inactive
  - slot eligibility
  - surfaces affected
- item detail should explain:
  - where the item can be applied
  - whether it changes character/status, room/world, profile/showcase, or multiple surfaces
  - whether it is passive, equip-only, display-only, or multi-use
- when an item is equipped/applied, the user should get clear confirmation and visible result
- keep this premium and readable

==================================================
H. CONSISTENCY / VALIDATION / ANTI-ERROR RULES
==================================================

Because item application will connect many systems, validation must be strong.

Requirements:
- server-side validation for all equip/application actions
- reject invalid slot/category/surface combinations
- prevent equipping non-owned items
- prevent duplicate impossible active states
- preserve consistency between:
  - inventory
  - marketplace ownership
  - room/world display state
  - character/status visualization
  - profile/showcase state
- keep all action responses JSON-only
- preserve ownership checks

If needed:
- add small repair/recompute logic so world/status/profile state can be reconciled from item application state

==================================================
I. MINIMAL ADMIN / INTERNAL CONTROLS
==================================================

Add only minimal internal controls if strictly needed.

Possible examples:
- item application eligibility mapping
- slot eligibility flags
- visual influence classification
- recompute/rebuild application state for a user if corrupted

Requirements:
- admin-only
- lightweight
- use existing admin/tuning surfaces where possible
- avoid building a large item-logic console

K. BACKEND / SECURITY RULES
==================================================

- preserve JSON-only APIs
- preserve ownership checks

- keep schema changes additive and minimal
- preserve existing mission/proof/reward/economy/world/community/premium flows
- keep applied/equipped/display state recomputable and internally trustworthy
